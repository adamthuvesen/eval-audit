# Bring-your-own-data minimal example

This is a worked walkthrough of the `eval-audit` input contract. The toy fixture
in this directory demonstrates a 2-agent 10-task audit where `alice` (8/10 = 80%)
clearly outperforms `bob` (4/10 = 40%) at twice the per-task cost. The audit's
verdict is unambiguous: `switch` — claim is supported and the effect favours
the treatment. See the rendered audit at
[`reports/byo-minimal/report.md`](../../reports/byo-minimal/report.md).

## What's in this directory

- [`study.yaml`](study.yaml) — minimal `StudySpec` declaring a single
  treatment-vs-control claim with `target_mde: 0.05`.
- [`make_runs.py`](make_runs.py) — Python script that constructs `runs.parquet`
  from inline data. **This is the BYO construction pattern most users will
  follow:** keep your data definition in code, write `runs.parquet` from it.
  Re-running the script overwrites `runs.parquet` — keep your edits in the
  script, not in the parquet directly.
- `runs.parquet` — 20 canonical-RunRecord-shaped rows generated by the script.
- `README.md` — this file.

## How to read the rendered audit

The rendered report at
[`reports/byo-minimal/report.md`](../../reports/byo-minimal/report.md) opens
with the v1 audit summary: a verdict (`switch`), a claim status
(`supported`), a one-line "why" with the bootstrap CI and cost ratio, a
"what would change it" line driven by resolution planning, and a reviewer-
pushback line. The Robustness Review table later in the report shows whether
the verdict survives standard reviewer perturbations across five fixed
dimensions.

## Adapting this to your data

1. **Edit `make_runs.py`.** Replace `ALICE_SUCCESS` and `BOB_SUCCESS` with your
   per-task outcomes. Replace `agent_id` strings with your agent identifiers.
   Replace `cost`, `tokens_in`, `tokens_out` values with your real numbers.
2. **Run the script:** `python make_runs.py` regenerates `runs.parquet`.
3. **Edit `study.yaml`** so `claims[].treatment` and `claims[].control` match
   your agent identifiers, and adjust `inference.target_mde` to the smallest
   effect you'd want to detect.
4. **Run the audit:**

   ```bash
   eval-audit validate runs.parquet study.yaml
   eval-audit check study.yaml --runs runs.parquet
   eval-audit analyze study.yaml --runs runs.parquet
   eval-audit report  study.yaml --runs runs.parquet --skip-validation
   ```

For the formal field-by-field `RunRecord` reference (every column, what
analysis uses it for, BYO guidance for awkward fields), see
[`agents/docs/INPUT_CONTRACT.md`](../../agents/docs/INPUT_CONTRACT.md).

## Why this example exists

Every `eval-audit` user eventually ends up writing something like
`make_runs.py`. Treating it as a worked example up front — with the script,
the parquet, the study declaration, and the rendered audit all committed
together — makes the input contract concrete instead of abstract.
